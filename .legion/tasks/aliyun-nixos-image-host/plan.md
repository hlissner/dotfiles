# Aliyun NixOS Image Host

## Goal
Add a dedicated NixOS host configuration and image build entry for producing an Alibaba Cloud ECS-importable NixOS system image from this dotfiles repository.

## Problem
The existing `hosts/acorn/azure-image` path builds an Azure-specific VHD using Azure NixOS modules. Alibaba Cloud ECS does not have an upstream nixpkgs `aliyun-image.nix` module, but ECS can import generic QCOW2/RAW/VHD/VMDK images from OSS. We need a host/image path that is cloud-generic enough for ECS import while preserving the repository's host/module style.

## Acceptance
- A new host exists under `hosts/` and is visible in `.#hostMetadata` / `.#nixosConfigurations`.
- The host avoids Azure-specific modules and uses generic QEMU/cloud image plumbing suitable for ECS.
- A build entry exists to produce an ECS-importable disk image, preferably QCOW2.
- `nix build` for the new image evaluates/builds successfully, or any build blocker is explicitly documented with evidence.
- Existing unrelated dirty work in the main checkout is not touched.

## Scope
- Add a new `hosts/aliyun-acorn` NixOS server host.
- Add a nested image flake under that host for `nix build ./hosts/aliyun-acorn/image#aliyun-image`.
- Include minimal ECS boot/network/init compatibility: QEMU guest profile, cloud-init, DHCP-capable network config, serial console, grow partition, ext4 root and EFI boot partition labels.
- Reuse existing dotfiles profiles and C1 SSH key configuration.

## Non-goals
- Do not upload to OSS or call Alibaba Cloud APIs in this task.
- Do not replace or mutate the existing `acorn` Azure image path.
- Do not change secrets or deploy credentials.
- Do not tune production services beyond the existing acorn-like server baseline.

## Assumptions
- Alibaba Cloud ECS import will accept QCOW2 generated by NixOS `virtualisation/disk-image.nix`.
- ECS metadata is compatible with cloud-init's AliYun datasource in packaged cloud-init, or at minimum DHCP/network comes up without provider-specific scripts.
- First iteration targets x86_64 Linux ECS.

## Constraints
- Follow Legion workflow and use an isolated git worktree.
- Keep persistent artifacts inside the repository.
- Preserve the existing main checkout dirty change in `hosts/charlie/default.nix`.

## Risks
- ECS may require provider-specific initialization beyond generic cloud-init; first boot must be validated on an actual ECS instance later.
- UEFI/BIOS boot mode mismatch can make an imported image unbootable; this iteration uses EFI and documents that import must match.
- Building a full image can be large and slow on the local machine.

## Design Summary
Use a new `aliyun-acorn` host instead of modifying `acorn`. The host mirrors acorn's server/dev/shell/service profile but replaces Azure modules with generic QEMU/cloud-init settings. A nested image flake references the parent dotfiles flake and extends `nixosConfigurations.aliyun-acorn` with `virtualisation/disk-image.nix`, producing a QCOW2 image via `system.build.image`.

## Phases
1. Create host and image build entry.
2. Run `nix build` / evaluation gates until the image target passes or a real blocker is captured.
3. Review diff for scope and safety.
4. Record walkthrough and wiki/writeback evidence.
