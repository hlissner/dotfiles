# Axiom Caelestia Shell Migration

## Task Identity

- Name: Axiom Caelestia Shell Migration
- Task ID: `axiom-caelestia-shell-migration`
- Trigger: user requested removing the end4 configuration and installing `https://github.com/caelestia-dots/shell` according to its README.
- Product direction: Caelestia shell replaces the repository-managed end4 `ii` desktop path. Backward compatibility with end4 is not required.

## Goal

Make Axiom's Wayland desktop use Caelestia shell as the active Quickshell-based product shell, installed through the upstream Nix-supported path, while preserving this repository's NixOS ownership of host facts, Hyprland session startup, systemd user units, XDG config, and Darwin isolation.

## Problem

The current desktop truth is a large vendored end4 `ii` import plus Nix glue that was added to make that import usable. The user now wants end4 removed and Caelestia shell installed instead. Keeping the end4 source tree, its matugen/fuzzel assumptions, and `modules.desktop.quickshell` defaults would leave two shell products in the repository and make the new direction ambiguous. The durable architecture should avoid another unmanaged manual clone and use Caelestia's upstream flake package/CLI integration as the source of the shell runtime.

## Acceptance Criteria

- Axiom's active shell service starts Caelestia shell, not `quickshell --config ii` or any legacy `axiom-shell` path.
- The flake declares `caelestia-dots/shell` as an input and uses the upstream Nix package with CLI support for the Linux desktop.
- The repository no longer links or vendors the end4 `config/quickshell/ii`, end4 matugen/fuzzel shell support, or end4-specific Hyprland shell override files as active desktop config.
- Hyprland remains Nix-owned: monitors, workspaces, XKB facts, default apps, UWSM/greetd/portal startup, session target, and host fallback keybinds stay generated by this repository.
- Caelestia configuration lives under repository/Nix-owned XDG config such as `~/.config/caelestia/shell.json` and optional CLI config, not unmanaged live-home edits.
- Darwin builds and shared shell/dev modules do not import Linux-only Caelestia or Hyprland concerns.
- Static validation proves the active service/package/config shape, absence of active end4 references, and Nix evaluation/build viability; live graphical validation is attempted only when a Hyprland user session is available.
- Legion design, verification, review, walkthrough, wiki writeback, and PR/worktree lifecycle evidence are recorded.

## Assumptions

- The target host is Axiom, a Linux Hyprland workstation, and the shell may not have a live Wayland session available during automation.
- Caelestia's README Nix path is preferable to manual installation because this repository is a NixOS flake and should not rely on mutable clones under `~/.config`.
- The upstream `with-cli` package is required because the README says the default package lacks CLI support needed for full functionality.
- Caelestia shell can own visual shell surfaces while this repository continues to own compositor/session integration and host-specific policy.
- Removing end4 source and docs is acceptable where they only represent the superseded product direction; historical Legion evidence may remain as past-task records until wiki closeout updates current truth.

## Constraints

- Do not run Caelestia or end4 mutable setup scripts.
- Do not vendor generated state, caches, secrets, wallpaper outputs, API keys, or live user config.
- Do not keep end4 as a fallback product path or preserve end4 UX compatibility.
- Do not let Caelestia/HM module assumptions take over repository-wide Home Manager architecture without an explicit design reason.
- Keep Linux desktop changes isolated from Darwin.
- Prefer deleting obsolete end4-specific integration over preserving dormant compatibility code.

## Scope

- Add the Caelestia shell flake input and expose a Linux-only package/config path for Axiom.
- Replace the end4-specific Quickshell module/service with a Caelestia shell integration module or equivalent NixOS-owned config.
- Simplify Hyprland checked-in config to a local, non-end4 base that sources Nix-generated host overrides.
- Remove end4 vendored shell/config directories and root-level end4 planning doc from active repository source.
- Generate Caelestia shell/CLI JSON config from Nix where local defaults are needed.
- Update Legion docs/wiki so current truth points to Caelestia, not end4.
- Run build/static validation and record live-session limits.

## Non-Goals

- Do not port end4-specific UI features, IPC names, Waffle components, or IllogicalImpulse assumptions onto Caelestia.
- Do not redesign Caelestia upstream UI or fork its source unless validation proves a small local patch is necessary.
- Do not migrate or preserve existing user-persisted end4 runtime settings.
- Do not solve unrelated Hyprland, theme, wallpaper, notification, clipboard, or AI/cloud policy issues beyond what is needed for Caelestia startup.
- Do not change non-Axiom hosts unless evaluation shows shared module breakage caused by this migration.

## Design Summary

- Treat Caelestia as an upstream packaged product shell and end4 as a superseded vendored product source.
- Use the upstream flake package `with-cli` rather than a manual clone or another source import, keeping updates and dependencies in `flake.lock`.
- Keep this repository's own NixOS module as the integration boundary: systemd user service, XDG config JSON, package exposure, reload hook, and any extra runtime tools that are truly host policy.
- Keep Hyprland base config small and local. Nix-generated files should remain the authoritative place for Axiom monitors, workspaces, apps, environment, input, rules, and fallback keybinds.
- Delete obsolete end4 directories instead of leaving inactive alternate shell products in-tree.
- Validate by checking the evaluated service command/package, generated config files, Hyprland config syntax where possible, absence of active end4 references, and an Axiom toplevel build.

## Risks

- Caelestia's upstream package may assume unstable dependencies or systemd/HM details that differ from this repository's custom Home Manager abstraction.
- Replacing a large Hyprland config can regress keybinds or rules if host-owned overrides are incomplete.
- Upstream Caelestia global shortcuts may require specific Hyprland `global` binds from the main Caelestia dots repository, so local fallback bindings may need careful mapping.
- A headless shell cannot prove panel rendering, tray icons, OSD, or lock/session UI behavior.
- Removing end4 source is a large diff and can expose unrelated references in old docs, tasks, or wiki that must be separated from active runtime references.
